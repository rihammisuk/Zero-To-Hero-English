Part 0 · Mindset

How to use this site

You don't get good at DSA by reading. You get good by recognising patterns — the same way a cook sees a recipe and instantly knows the technique. Most interview problems are old problems wearing a new shirt. Learn the techniques, and the shirts stop fooling you.

Restaurant analogy

A chef doesn't memorise 500 dishes. They master ~15 techniques — chop, sear, braise, emulsify — and combine them. DSA is the same: ~15 patterns cover ~90% of interview questions. This site teaches the techniques, not the dishes.

The one rule

For every problem, ask two questions before writing code: (1) What pattern is this? and (2) What's the time & space cost? If you can answer both, the code is the easy part.

The climb — 30 steps, 4 phases

Same ladder idea as your English course. You stand on the ground (basics), build the walls (structures), add the rooms (techniques), then the roof (advanced). Tick each step in the tracker at the bottom.

Part 0 · The plan

Your path: zero to offer

You don't need talent or a CS degree — you need the right order and steady reps. This is the whole journey on one page. If you've never written a loop, you start at the top and you don't skip. Follow it and you'll walk into a FAANG interview prepared.

Gym analogy

Nobody lifts 100kg on day one. You add a little each week and your body adapts. Coding interviews are identical — solve a few problems daily, and patterns that look impossible today feel obvious in two months. Consistency beats intensity.

How to practice so it actually sticks

The 20-minute rule

Try a problem alone for 20 minutes. Stuck? Read only the approach (not the full code), then close it and write the solution yourself from scratch. Reading solutions feels productive but teaches almost nothing — your hands have to do the work.

Learn patterns, not problems

There are thousands of problems but only ~15 patterns. After each problem ask: "which pattern was this, and what was the clue?" That one habit is what makes new problems feel familiar.

Re-solve on a schedule

Solve it today, again in 1 day, again in 1 week. The third time it's muscle memory. Tick it in the tracker only when you can solve it cold.

The 12-week plan

Twelve weeks if you practise an hour a day. Slower is completely fine — the order matters far more than the speed. Each block builds on the last.

Step 01 · Foundation

Big-O notation

Big-O answers one question: as the input grows, how fast does the work grow? It ignores small details and looks only at the shape of the growth.

GPS analogy

A GPS doesn't tell you "342 metres." It tells you "left, then straight." Big-O is the same — it ignores the exact number of steps and tells you the shape: does the trip double when the city doubles, or stay roughly the same?

The growth ladder — memorise this order

Big-ONamen=1,000 means…Verdict
O(1)Constant1 stepPerfect
O(log n)Logarithmic~10 stepsExcellent
O(n)Linear1,000 stepsGood
O(n log n)Linearithmic~10,000 stepsFine (sorting)
O(n²)Quadratic1,000,000 stepsSlow — avoid if possible
O(2ⁿ)ExponentialastronomicalOnly tiny inputs
How to read code

One loop over the input → O(n). A loop inside a loop → O(n²). Cutting the problem in half each time → O(log n). A hash-map lookup → O(1).

# O(n) — one pass
for x in arr:            # n times
    total += x

# O(n^2) — loop inside loop
for i in arr:            # n times
    for j in arr:        # n times each = n*n
        check(i, j)
Common trap

Two separate loops are O(n + n) = O(n), NOT O(n²). Nesting multiplies; sequencing adds.

How to calculate it — 3 rules
Drop constants
O(2n) → O(n). Doubling the work doesn't change the shape. We care about growth, not exact counts.
Drop smaller terms
O(n² + n) → O(n²). As n grows, the biggest term dwarfs the rest. Keep only the dominant one.
Multiply nesting, add sequencing
A loop inside a loop → multiply (n × n). Two loops one after another → add (n + n).
Time vs space — the two branches

Every solution has two Big-O numbers. Interviewers ask for both.

Time complexity
How the number of operations grows. "Will it finish before the timeout?"
Space complexity
How much extra memory you use beyond the input — arrays, hash maps, and the recursion call stack all count.
Trade-off: a hash map often buys O(n)→O(1) time by spending O(n) space. Naming that trade is a senior signal.
Best / average / worst — the case branches

The same algorithm can have different costs depending on the input. We usually quote the worst case, but know all three.

Best
The luckiest input. Quicksort on an already-good pivot, or finding your target on the first check.
Average
Typical random input. What you'll see in practice (hash map lookup is "average O(1)").
Worst
The nastiest input. Quicksort's O(n²) on a bad pivot; hash map's O(n) if everything collides. Quote this by default.
Two more you'll hear

Amortised O(1) — usually instant, occasionally expensive, cheap on average (appending to a dynamic array: rare resizes spread out to O(1) each). Recursion space — each recursive call sits on the call stack, so a recursion d levels deep costs O(d) space even if it returns nothing.

Steps 02–03 · Foundation

Arrays & strings

An array is a row of numbered boxes side by side in memory. Because they're numbered, you can jump to box #5 instantly — that's why reading by index is O(1).

Parking lot analogy

Numbered parking spots. Spot 42? Walk straight there — no searching (O(1) access). But to insert a car between spot 5 and 6, every car after must shift over (O(n) insert).

OperationCostWhy
Read arr[i]O(1)Jump straight to the index
Append at endO(1)*Amortised — usually free
Insert / delete in middleO(n)Everything after must shift
Search (unsorted)O(n)Check every box

Strings are arrays of characters with one catch: in many languages they're immutable — every "edit" builds a new string. Editing a string in a loop can secretly become O(n²). Build a list of chars and join once at the end.

Interview trap

Reversing a string with s = c + s in a loop is O(n²). Use a list + reverse, or two pointers, for O(n).

The branches
Static array
Fixed size, set once. Classic in C/Java. Fast and memory-tight, but can't grow.
Dynamic array append O(1)*
Grows automatically — Python list, Java ArrayList, JS array. Doubles in size when full (that's the amortised cost).
2-D array / matrix access O(1)
A grid — rows × cols. Used for boards, images, DP tables. Access with grid[r][c].
String
An array of characters, usually immutable. Treat edits as "build new," not "change in place."
Subarray vs subsequence — know the difference

Interviewers use these words precisely, and they change the whole approach.

Subarray
Contiguous — a connected slice. From [1,2,3,4]: [2,3] yes, [1,3] no. → think sliding window.
Subsequence
Keeps order but can skip. From [1,2,3,4]: [1,3] yes. → think two pointers or DP.
Subset
Any selection, order doesn't matter. → think backtracking.
Must-know array techniques
Prefix sums query O(1)
Pre-add the array so any range sum is one subtraction. Turns repeated range queries from O(n) each into O(1).
Use when: "sum between i and j" asked many times.
In-place swap O(1) space
Rearrange using the array itself — reverse, rotate, Dutch-flag — without a second array.
Two pointers O(n)
Two indices replace a nested loop (see its own section).
Sliding window O(n)
A moving range for "best contiguous stretch" problems.
Step 04 · Foundation

Hash maps & sets

This is the tool you'll reach for more than any other. A hash map stores key → value pairs and finds any key in O(1) — instantly, no matter how big it gets.

Library analogy

A library catalogue. You don't walk every shelf — you look up the book's code and go straight there. The hash function is the catalogue: it turns a key into an address.

The pattern to remember

Whenever a problem says "have I seen this before?" or "how many times does X appear?" or "find the pair that sums to…" — reach for a hash map. It turns an O(n²) search into O(n).

# Two Sum — the classic. Brute force is O(n^2).
# Hash map makes it O(n):
def two_sum(nums, target):
    seen = {}                 # value -> index
    for i, x in enumerate(nums):
        need = target - x
        if need in seen:       # O(1) lookup
            return [seen[need], i]
        seen[x] = i
    return []
▶ See it work
nums = [2, 7, 11, 15], target = 9. We walk left to right, remembering what we've seen.
At 2: we need 7 to reach 9. Have we seen a 7? No. Remember 2.
271115seen = {2}
At 7: we need 2. Have we seen a 2? Yes! Done.
271115answer = [0, 1]
One pass, every lookup instant. O(n) — instead of checking every pair, O(n²).

A set is the same idea without values — just "is this thing in here?" Use it for de-duplication and fast membership checks.

How it works inside (so you can explain it)

A hash map is really an array of buckets. A hash function turns your key into a number, that number picks a bucket, and the pair is stored there. Reading does the same maths and jumps straight to the bucket — that's the O(1).

Hash function
Turns any key (string, number) into a bucket index. A good one spreads keys evenly.
Collision
Two keys land in the same bucket. Handled by chaining (a little list per bucket) or open addressing (probe to the next free slot).
Load factor
How full the table is. When it gets high, the map resizes and re-hashes everything — the reason append is amortised O(1).
When O(1) becomes O(n)

If every key collides into one bucket, lookups walk a list and degrade to O(n). Rare in practice, but it's why hash maps are "average O(1), worst O(n)." Also: keys must be immutable/hashable (numbers, strings, tuples — not lists).

The branches
Hash map / dict O(1)
Key → value. The workhorse. {} in Python, HashMap in Java.
Hash set O(1)
Keys only — "have I seen this?" De-dup and membership.
Counter / multiset O(1)
Key → count. Frequency problems, anagrams. Python Counter.
Ordered / sorted map O(log n)
Keeps keys sorted (tree-based, e.g. Java TreeMap). Slower, but supports "next bigger key" and range queries.
Steps 12–13 · Techniques

Two pointers & sliding window

Two pointers means walking through data with two indexes instead of one. It collapses many O(n²) problems to O(n).

Two-hands analogy

Reading a sorted list with both hands — one at the start, one at the end. If the pair is too big, move the right hand in; too small, move the left hand out. You meet in the middle in one pass.

# Pair that sums to target in a SORTED array
def pair_sum(arr, target):
    lo, hi = 0, len(arr) - 1
    while lo < hi:
        s = arr[lo] + arr[hi]
        if s == target: return (lo, hi)
        if s < target:  lo += 1   # need bigger
        else:           hi -= 1   # need smaller
▶ See it work
arr = [1, 3, 4, 6, 8, 11], target = 10. One hand at each end.
13468111+11 = 12 > 10 → move right in
Right hand → 8.
13468111+8 = 9 < 10 → move left out
Left hand → 3.
13468113+8 = 11 > 10 → move right in
Right hand → 6.
13468113+6 = 9 < 10 → move left out
Left hand → 4. Found it.
13468114 + 6 = 10 ✓
The hands never backtrack — each moves inward once. O(n), no nested loop.

Sliding window

A window is a sub-section of the array that grows and shrinks as you slide it across. Perfect for "longest/shortest sub-array or substring that satisfies…" questions.

Recognise it

Keywords: "contiguous", "substring", "subarray", "longest", "at most K". Expand the window with the right edge; when a rule breaks, shrink from the left.

▶ See it work
"abcabcbb" — longest stretch with no repeated letter. Grow the window right; if a letter repeats, shrink from the left until it's gone.
Window grows: a, b, c — all unique. Length 3.
abcabcbbbest = 3
Right edge hits a — but "a" is already in the window. Shrink the left edge past the old "a".
abcabcbbwindow = "bca", still 3
It keeps sliding the same way to the end. The answer is the longest the window ever reached: 3.
Each letter is added once and removed at most once → O(n).
The four variants — pick by the clue
Opposite ends O(n)
One pointer at each end, moving inward. Needs sorted data usually.
Use for: pair sum, palindrome check, container with most water, reverse in place.
Fast & slow O(n)
Both start left; one moves faster. A "reader" and a "writer."
Use for: remove duplicates in place, cycle detection, find the middle.
Fixed window O(n)
Window of a set size K slides along; add the new item, drop the old.
Use for: max sum of K items, averages of every K.
Variable window O(n)
Grow right to satisfy a rule, shrink left when it breaks.
Use for: longest substring no repeats, min window, "at most K distinct."
The deciding question

Is the data sorted and you want a pair? → opposite ends. Is it about a contiguous stretch? → window. Is it "in-place rearrange"? → fast & slow.

Steps 05–06 · Structures

Stacks & queues

Both are just lists with a rule about which end you use.

Everyday analogy

Stack = a pile of plates. You add and take from the top — last in, first out (LIFO). Queue = a line at the shop. First person in is first served (FIFO).

StructureRuleUse it for
Stack (LIFO)push/pop the topMatching brackets, undo, DFS, "next greater element"
Queue (FIFO)add back, remove frontBFS, scheduling, level-order traversal
Classic stack problem

Valid parentheses: push every opening bracket; on a closing bracket, the top of the stack must be its match. Empty stack at the end = valid. Pure O(n).

Operations — all O(1)
Stack
push (add to top), pop (remove top), peek (look at top). Built from a dynamic array or linked list.
Queue
enqueue (add to back), dequeue (remove front), front (look). Use a deque — a plain array's pop-from-front is O(n)!
The branches
Stack (LIFO) O(1)
Last in, first out. Brackets, undo, DFS, expression eval.
Queue (FIFO) O(1)
First in, first out. BFS, scheduling, level-order.
Deque O(1)
Double-ended — add/remove at both ends. Powers the sliding-window-maximum trick.
Monotonic stack O(n)
A stack kept always increasing or decreasing. Finds the nearest greater/smaller element in one pass.
Use for: daily temperatures, next greater element, largest rectangle.
Priority queue O(log n)
Not FIFO — always serves the highest priority next. That's a heap (own section).
Circular queue O(1)
A fixed array reused in a loop — efficient for buffers and streaming.
Step 07 · Structures

Linked lists

A linked list is a chain: each node holds a value and a pointer to the next node. No numbered boxes — to reach node #5 you must walk 1→2→3→4→5.

Treasure-hunt analogy

Each clue tells you where the next clue is. You can't skip ahead — but inserting a new clue is easy: just rewrite one pointer. Arrays are the opposite: instant access, painful insert.

The two tricks that cover most problems

1. Reversal — flip the pointers one by one. 2. Fast & slow pointers — one moves 2 steps, one moves 1. Used to find the middle, or detect a cycle (if they ever meet, there's a loop).

# Reverse a linked list — memorise this shape
def reverse(head):
    prev = None
    while head:
        nxt = head.next   # save next
        head.next = prev  # flip pointer
        prev = head       # advance prev
        head = nxt        # advance head
    return prev
Linked list vs array — the trade
OperationLinked listArray
Access by indexO(n) walk the chainO(1) jump
Insert / delete at a known nodeO(1) rewire pointersO(n) shift everything
SearchO(n)O(n)
Memoryextra pointer per nodecompact, contiguous
The branches
Singly linked
Each node points to the next only. Lightest; you can only walk forward.
Doubly linked
Points to next and previous. Walk both ways and delete a node in O(1). Powers the LRU cache.
Circular
The tail points back to the head — a loop. Used for round-robin scheduling and buffers.
The techniques that crack most problems
Dummy head node
A fake node before the real head, so you never special-case "empty list" or "delete the first node." Used in nearly every list problem.
Pointer reversal
Flip next pointers one at a time with prev/curr/next.
Fast & slow (Floyd's)
One pointer 2×, one 1×. Finds the middle; if they meet, there's a cycle.
Merge / weave
Splice two lists together node by node with a dummy head (merge sorted lists, reorder list).
Step 14 · Techniques

Binary search

If data is sorted, you never check every item. You check the middle, throw away half, repeat. That's how 1,000 items take only ~10 checks — O(log n).

Dictionary analogy

Looking up "moon" in a paper dictionary — you don't start at "aardvark." You open the middle, see "M" is ahead, and jump. Each open halves what's left.

def binary_search(arr, target):
    lo, hi = 0, len(arr) - 1
    while lo <= hi:
        mid = (lo + hi) // 2
        if arr[mid] == target: return mid
        if arr[mid] < target:  lo = mid + 1
        else:                  hi = mid - 1
    return -1
▶ See it work
Find 7 in [1, 3, 5, 7, 9, 11]. Check the middle, throw away the half that can't contain it.
Middle is 5. 7 > 5, so 7 must be on the right. Discard the left half.
1357911look right of 5
New middle of the right half is 9. 7 < 9, so go left. Discard 9 and 11.
1357911look left of 9
Only 7 is left. Found it.
13579113 checks, not 6
Each check halves what's left. 1,000 items → ~10 checks. That's O(log n).
The hidden superpower

Binary search isn't only for sorted arrays. Any time the answer lives in a numeric range and you can ask "is X good enough? (yes/no)", you can binary-search the answer itself — "minimum capacity", "smallest speed", etc.

The branches
Exact match O(log n)
The basic version above — find a value, or report it's absent.
Lower / upper bound O(log n)
Find the first or last position — used when there are duplicates, or for "insert position." Don't stop on a match; keep narrowing.
Use for: first/last occurrence, count of a value.
Binary search on the answer O(n log m)
Search a numeric range, not an array, using a yes/no feasibility check.
Use for: Koko bananas, ship capacity, "minimum largest sum."
Rotated sorted array O(log n)
One half is always still sorted — figure out which, then decide your side.
2-D matrix O(log mn)
A sorted grid treated as one long sorted line via index maths.
The thing that breaks binary search

Off-by-one bugs and infinite loops. Pin down three things every time: is hi the last index or one past it? Is the loop lo <= hi or lo < hi? Does each branch actually shrink the range? Pick one template (see Templates) and reuse it — don't improvise.

Steps 16–17 · Techniques

Recursion & backtracking

Recursion is a function that calls itself on a smaller piece of the problem, until the piece is tiny enough to answer directly.

Russian-dolls analogy

Open a doll, find a smaller doll, open that one… until you reach the tiny solid one (the base case). Then you close them all back up (the answers combine on the way out).

Every recursion needs two things

1. Base case — when to stop (or it loops forever). 2. Recursive step — call yourself on something smaller. Always write the base case first.

Backtracking = recursion that undoes

Try a choice, recurse, and if it fails, undo the choice and try the next one. This is how you generate permutations, combinations, subsets, and solve mazes / sudoku / N-queens.

# Backtracking skeleton — every such problem fits this
def backtrack(path, choices):
    if is_solution(path):
        results.append(path[:])   # save a copy
        return
    for c in choices:
        path.append(c)            # choose
        backtrack(path, next_choices)  # explore
        path.pop()                # un-choose (the key step)
▶ See it work
factorial(3) = 3 × 2 × 1. Real-life: nested gift boxes. You can't know the total until you open down to the smallest box, then wrap back up.
Going down (opening the boxes). Each call waits on a smaller one:
fact(3)= 3 × fact(2) … waiting
↳fact(2)= 2 × fact(1) … waiting
↳↳fact(1)base case = 1, stop!
Coming back up (re-wrapping). Now each waiting call gets its answer:
fact(1)returns 1
fact(2)2 × 1 = 2
fact(3)3 × 2 = 6 ✓
The base case (fact(1)=1) is what stops it. Forget it and the boxes never end — that's an infinite loop. Backtracking is this same down-and-up, but you also undo your choice on the way back up.
Anatomy & cost
Base case
The stopping condition. Write it first. No base case = stack overflow.
Recursive step
Call yourself on a smaller input that moves toward the base case.
Space cost
Each pending call sits on the call stack → depth d costs O(d) memory, even with no extra arrays.
Kinds of recursion
Linear
One call per step (factorial, list traversal). Depth = n.
Tree / branching
Two+ calls per step (Fibonacci, tree traversal, subsets). This is where work can explode — and where memoisation rescues you.
Memoised → DP
Cache each answer so a sub-problem is solved once. This is literally the bridge from recursion to dynamic programming.
Backtracking families — same skeleton, different "choices"
Subsets O(2ⁿ)
Each element is either in or out. The power set.
Permutations O(n!)
All orderings. Track which elements are already used.
Combinations O(2ⁿ)
Pick k of n, order doesn't matter. Pass a start index so you never go backward.
Grid / constraint varies
Word search, N-queens, sudoku — place, recurse, undo if it fails. Pruning early (abandon dead branches) is the key to speed.
Step 08 · Structures

Trees & binary search trees

A tree is data that branches: one root at the top, each node pointing to children below. It's recursion made visible — almost every tree problem is solved by recursion.

Family-tree analogy

A grandparent at the top, children branching down. To count everyone, you count yourself + everyone in the left branch + everyone in the right branch. That sentence is the recursive solution.

TraversalOrderGood for
Inorderleft → node → rightBST gives sorted output
Preordernode → left → rightCopying / serialising a tree
Postorderleft → right → nodeDeleting, computing from leaves up
Level-order (BFS)row by rowShortest path, "by level" questions
Binary Search Tree rule

In a BST, everything on the left is smaller, everything on the right is bigger. That means search, insert, and delete are O(log n) when balanced — like binary search, but on a tree.

Words interviewers use
Root / leaf
Root = top node. Leaf = a node with no children (the ends).
Height / depth
Height = longest path root→leaf. A balanced tree has height ≈ log n; a degenerate one ≈ n.
Balanced
Left and right heights stay close, so operations stay O(log n). The opposite (a straight line) is O(n).
Two ways to walk a tree

DFS (depth-first) dives down one branch fully before the next — it comes in three orders (inorder, preorder, postorder) and is naturally recursive. BFS (level-order) sweeps row by row using a queue. The table above shows when each order is useful.

The branches — the whole tree family
Binary tree
Each node has ≤ 2 children. The base of everything here.
Binary search tree (BST) O(log n)*
Left < node < right. Ordered search/insert/delete — when balanced.
Balanced BST (AVL, Red-Black) O(log n)
Self-rotates to stay short, guaranteeing O(log n). What real "tree maps" use under the hood.
Heap O(log n)
A complete tree where parent beats child. Its own section — used for priority.
Trie (prefix tree) O(L)
Branches by character, for fast word/prefix lookup (autocomplete, spell-check).
N-ary tree
Any number of children — file systems, the DOM, org charts.
Segment / Fenwick tree O(log n)
Advanced: range sum/min queries with updates. Rare in interviews, good to name.
The BST catch

A plain BST is only O(log n) if it stays balanced. Insert already-sorted data and it becomes a straight line — effectively a linked list, O(n). That's why production code uses self-balancing trees.

Step 09 · Structures

Heaps / priority queues

A heap is a structure that always keeps the smallest (or largest) item ready at the top, so you can grab it in O(1) and re-balance in O(log n).

Hospital analogy

A hospital triage queue. Patients aren't seen in arrival order — the most urgent is always pulled next, no matter when they arrived. The heap keeps "most urgent" at the front automatically.

Recognise it

Keywords: "top K", "K largest/smallest", "median of a stream", "merge K sorted lists". A heap of size K solves "top K" in O(n log K) instead of sorting everything in O(n log n).

Operations
peek
O(1) — look at the top (min or max) without removing it.
push
O(log n) — add an item and let it "bubble up" to its spot.
pop
O(log n) — remove the top and let the new top "sink down."
heapify
O(n) — turn a whole array into a heap in one pass (faster than n pushes).
Quietly clever

A heap is stored as a plain array, not nodes with pointers. For index i, children sit at 2i+1 and 2i+2. That's why it's so memory-efficient.

The branches
Min-heap
Smallest on top. Default in Python (heapq). For "K largest," push and pop the smallest.
Max-heap
Largest on top. For "K smallest." In Python, push negated values.
Priority queue
The same thing by another name — serve highest priority next. Powers Dijkstra and schedulers.
Two-heap pattern
A max-heap for the low half + a min-heap for the high half, kept balanced → the running median of a stream in O(log n).
Steps 11, 18–19 · Structures & Techniques

Graphs

A graph is nodes connected by edges — the most flexible structure, because almost everything real is a graph: maps, friend networks, web pages, course prerequisites.

City-map analogy

Cities are nodes, roads are edges. "Shortest route" = graph search. A tree is just a graph with no loops and one path between any two points.

BFS vs DFS — the two ways to explore

SearchHow it movesUsesBest for
BFSlevel by level, ripples outa queueShortest path (unweighted)
DFSdeep down one path, then backa stack / recursionCycles, connectivity, all paths
The graph toolkit

Union-Find — are two nodes connected? Topological sort — order tasks with dependencies (course schedule). Dijkstra — shortest path when edges have weights (a heap + BFS).

Never forget

Always track visited nodes. Without it, a graph with a loop sends your search into an infinite cycle.

Types of graph — name them out loud
Directed
Edges have a direction (Twitter follows). vs undirected — mutual (Facebook friends).
Weighted
Edges carry a cost/distance (road lengths). vs unweighted — every hop equal.
Cyclic / DAG
Has loops, or is a DAG (directed, no cycles) — the shape that allows topological ordering.
Two ways to store one
Adjacency list O(V+E) space
Each node keeps a list of its neighbours. The default — efficient for sparse graphs (most real ones).
Adjacency matrix O(V²) space
A grid where cell [i][j] says if i→j exists. O(1) edge lookup, but heavy. Good for dense graphs.
The algorithm family — the real branches
BFS O(V+E)
Ripples out level by level (queue). Shortest path on an unweighted graph.
DFS O(V+E)
Dives deep then backtracks (stack/recursion). Connectivity, cycle detection, all paths.
Dijkstra O(E log V)
Shortest path with non-negative weights. BFS + a min-heap.
Bellman-Ford O(V·E)
Shortest path that allows negative edges; also detects negative cycles. Slower than Dijkstra.
Floyd-Warshall O(V³)
Shortest path between every pair of nodes. Tiny graphs only.
Topological sort O(V+E)
Orders a DAG so prerequisites come first (course schedule, build order). Also detects cycles.
Union-Find ~O(1)
Are two nodes connected? Counts components, detects cycles in undirected graphs.
MST: Prim / Kruskal O(E log V)
Cheapest set of edges that connects everything (network/cable layout). Kruskal uses Union-Find.
How to choose, fast

Unweighted shortest path → BFS. Weighted shortest path → Dijkstra (or Bellman-Ford if negatives). Ordering with prerequisites → topological sort. Grouping / "connected?" → Union-Find. Just explore everything → DFS.

Step 20 · Techniques

Sorting

In an interview you'll usually call the built-in sort (it's O(n log n) and battle-tested). But you must understand how the fast ones work, because the ideas reappear everywhere.

AlgorithmTimeIdea in one line
Bubble / insertionO(n²)Swap neighbours — only for tiny / nearly-sorted data
Merge sortO(n log n)Split in half, sort each, merge — stable, predictable
Quick sortO(n log n)*Pick a pivot, partition around it — fast in practice
Counting / radixO(n)No comparisons — only for bounded integers
The takeaway

Merge sort teaches "divide and conquer." Quick sort teaches "partition around a pivot" — the same partition trick finds the K-th largest element in O(n). Learn the ideas, reuse the built-in.

Two properties interviewers ask about
Stable
Equal items keep their original order. Matters when sorting by one field but preserving another (sort by age, keep name order). Merge sort is stable; quicksort isn't.
In-place
Sorts using O(1) extra space (no second array). Quicksort and heapsort are in-place; merge sort needs O(n) extra.
The branches — comparison sorts
Bubble / selection O(n²)
Repeatedly swap or pick the smallest. Teaching tools — too slow for real use.
Insertion O(n²)
Build the sorted part one item at a time. Genuinely fast on small or nearly-sorted data.
Merge sort O(n log n)
Split, sort halves, merge. Stable and predictable; needs O(n) space.
Quick sort O(n log n)*
Partition around a pivot. Fastest in practice, in-place — but O(n²) worst case on bad pivots.
Heap sort O(n log n)
Build a heap, pop repeatedly. In-place, guaranteed — but not stable.
Non-comparison sorts — beat O(n log n)
Counting sort O(n+k)
Tally each value, rebuild. Only for integers in a small known range.
Radix sort O(n·d)
Sort by digit, least-significant first. For fixed-width integers/strings.
Bucket sort O(n)*
Spread into buckets, sort each. Good for evenly-spread floats.
Why O(n log n) is usually the floor

Any sort that works by comparing elements can't beat O(n log n) — that's a proven limit. The O(n) sorts cheat by not comparing (they bucket by value), which only works when values are bounded.

Steps 22–23 · Mastery

Dynamic programming

DP scares people, but it's one idea: don't solve the same sub-problem twice — remember the answer. That's it. Recursion that keeps a notebook.

Stairs analogy

"How many ways to climb 10 stairs taking 1 or 2 steps?" To answer step 10, you need steps 9 and 8. To answer 9, you need 8 and 7. Write each answer down once, and the whole thing is O(n) instead of exponential.

Spot a DP problem

Keywords: "how many ways", "minimum/maximum cost", "longest …", "can you reach…", AND the answer to a big case is built from smaller cases. That overlap is the signal.

# Climbing stairs — the "hello world" of DP
def ways(n):
    dp = [0] * (n + 1)
    dp[0] = dp[1] = 1
    for i in range(2, n + 1):
        dp[i] = dp[i-1] + dp[i-2]   # reuse, don't recompute
    return dp[n]
▶ See it work
How many ways to climb 5 stairs, 1 or 2 at a time? Real-life: you're standing at the bottom counting routes to the top. We build a notebook of answers for 0,1,2,… stairs and never recompute.
Base cases: 0 stairs = 1 way (do nothing). 1 stair = 1 way.
11····dp[0]=1, dp[1]=1
To reach stair 2, your last move came from stair 1 or stair 0. So ways(2) = ways(1) + ways(0) = 2.
112···1 + 1 = 2
Stair 3 = ways(2) + ways(1) = 2 + 1 = 3. Just add the two cells behind it.
1123··2 + 1 = 3
Stair 4 = 3 + 2 = 5. Stair 5 = 5 + 3 = 8. Each cell is just its two neighbours added.
112358answer = 8 ways
We filled the notebook once, left to right. O(n) — no stair counted twice. That's all DP is.
How to attack any DP

1. Define the state (what does dp[i] mean?). 2. Write the recurrence (how does dp[i] use earlier values?). 3. Set the base case. 4. Decide the fill order. Nail those four and the code writes itself.

Two ways to write any DP
Top-down (memoisation)
Write the natural recursion, then cache each answer. Easier to think up; uses recursion-stack space.
Bottom-up (tabulation)
Fill a table from the smallest cases up (the stairs example). No recursion, often less memory. Usually the interviewer's preference.
The DP families — most problems are one of these
1-D sequence O(n)
Answer depends on a few earlier indices. Climbing stairs, house robber, max subarray.
2-D grid O(m·n)
Paths through a grid, summing from top/left. Unique paths, min path sum, maximal square.
Knapsack O(n·W)
Choose items under a budget. 0/1 (each item once) vs unbounded (reuse) — coin change is unbounded knapsack.
Two-string O(m·n)
A grid over two strings. Longest common subsequence, edit distance.
Interval DP O(n³)
Combine ranges. Matrix-chain, burst balloons.
DP on trees / bitmask varies
State carried over a tree, or a bitmask tracking a set (travelling salesman for small n).
Step 21 · Mastery

Greedy

Greedy means: at each step, grab the best-looking option right now and never look back. Fast and simple — but only correct when local best leads to global best.

Cashier analogy

Giving change with the fewest coins: always hand over the biggest coin that fits, then repeat. With normal currency this greedy choice is always optimal — but with weird coin sets it can fail, which is exactly when you need DP instead.

Greedy vs DP

If a problem has a clean "best choice now" that you can prove never hurts later → greedy (faster). If choices interact and you might need to reconsider → DP (safer). When unsure in an interview, say so out loud and test greedy on a small example.

How to know greedy is safe
Greedy-choice
Taking the best local option leaves an optimal solution still reachable. If you can argue this, greedy works.
Test it
Try to break it with a small counterexample. Can't find one after a few tries? Greedy is probably right. Found one? Switch to DP.
Often "sort first"
Most greedy solutions begin by sorting (by end time, by size, by ratio) — the sort reveals the order to be greedy in.
Classic greedy types
Interval scheduling
Sort by end time, keep the earliest-finishing. Non-overlapping intervals, meeting rooms.
Reach / jump
Track the farthest you can get. Jump game.
Merge / sweep
Sort by start, then sweep and combine. Merge intervals.
Huffman / heap-greedy
Repeatedly take the two smallest (a heap). Optimal merging, task scheduling.
The classic greedy failure

Coin change with coins [1, 3, 4] for amount 6: greedy grabs 4+1+1 (3 coins), but 3+3 (2 coins) is better. Local-best lost. That's the signal to use DP instead.

How a FAANG interview actually works

If you've never interviewed at a big tech company, the process can feel like a black box. It isn't. It's the same handful of stages every time — here's the whole journey, start to offer.

Airport analogy

Think of airport stages: check-in, security, boarding, then the flight. Each gate only checks one thing, and you clear them in order. The interview is the same — pass one stage, move to the next.

Recruiter callA friendly 30-minute chat about your background and the role. Not technical — just be normal, curious, and clear about what you've built.

Phone screen / online test1–2 coding problems on a shared screen, ~45–60 min. The first real filter. This is exactly what the problem bank below trains.
Week 1–2
The onsite "loop"4–6 back-to-back rounds (~45 min each): 2–4 coding, 1 behavioral, and — for senior roles — 1 system design. The main event.
Week 3–5
Team match & offerInterviewers compare notes (the "debrief"). The recruiter calls with the result — and yes, you can negotiate the offer.
Week 5–8

What each round is grading

RoundWhat they're really checking
CodingCan you spot the pattern, write clean correct code, and talk while you think? Use the 45-minute playbook below.
BehavioralAre you someone people want to work with? Teamwork, ownership, handling conflict and failure.

The behavioral round — tell a good story

Half of cracking the loop is the non-coding round, and almost nobody prepares for it. The trick is a simple story format called STAR.

Storytelling analogy

A good story has a setup, a problem, what the hero did, and how it ended. STAR is just that structure for your work stories, so you don't ramble.

S · T · A · R

Situation — set the scene. Task — what was your job. Action — what you did (say "I", not "we"). Result — how it ended, with a number if you can.

Worked example — "Tell me about a deadline you saved"

S: Our release was a week away and a payment bug was blocking it. T: I owned the checkout module, so fixing it was on me. A: I traced it to a race condition, wrote a focused fix and tests, and paired with QA to verify overnight. R: We shipped on time; the bug never recurred and checkout errors dropped to near zero.

Prepare 6 stories, reshape them live

You don't need 50 answers — just 6 strong stories you can bend to fit most questions: a conflict, a failure you learned from, a leadership moment, a tight deadline, a disagreement with a manager, and your proudest project.

Pattern templates

These are the reusable skeletons that solve entire families of problems. Memorise the shape, and in the interview you only adapt the few lines that change. This is the single highest-leverage thing on this site.

When: longest/shortest contiguous subarray or substring matching a condition ("at most K", "without repeats").

def window(s):
    left = 0; best = 0; state = {}
    for right in range(len(s)):
        state[s[right]] = state.get(s[right], 0) + 1   # expand
        while invalid(state):                            # shrink
            state[s[left]] -= 1
            left += 1
        best = max(best, right - left + 1)
    return best

When: "minimum/maximum X such that it works" — capacity, speed, days. You can't search the array, so you search the answer range.

def min_feasible(lo, hi):
    while lo < hi:
        mid = (lo + hi) // 2
        if feasible(mid):   # can we do it with 'mid'?
            hi = mid          # try smaller
        else:
            lo = mid + 1      # need bigger
    return lo

When: shortest path in an unweighted graph/grid, or anything "by level".

from collections import deque
def bfs(start):
    q = deque([start]); seen = {start}; steps = 0
    while q:
        for _ in range(len(q)):   # one level
            node = q.popleft()
            for nxt in neighbors(node):
                if nxt not in seen:
                    seen.add(nxt); q.append(nxt)
        steps += 1
    return steps

When: generate all subsets / permutations / combinations, or constraint puzzles (N-queens, sudoku, word search).

def backtrack(start, path):
    res.append(path[:])              # record (subsets)
    for i in range(start, len(nums)):
        path.append(nums[i])         # choose
        backtrack(i + 1, path)      # explore
        path.pop()                   # un-choose

When: "are these connected?", counting groups/components, detecting cycles in an undirected graph.

class DSU:
    def __init__(self, n):
        self.p = list(range(n))
    def find(self, x):
        while self.p[x] != x:
            self.p[x] = self.p[self.p[x]]   # path compression
            x = self.p[x]
        return x
    def union(self, a, b):
        self.p[self.find(a)] = self.find(b)

When: order tasks with prerequisites (course schedule, build order). Also detects cycles.

from collections import deque
def topo(n, edges):
    indeg = [0]*n; g = [[] for _ in range(n)]
    for a, b in edges:        # a before b
        g[a].append(b); indeg[b] += 1
    q = deque(i for i in range(n) if indeg[i]==0)
    order = []
    while q:
        x = q.popleft(); order.append(x)
        for y in g[x]:
            indeg[y] -= 1
            if indeg[y]==0: q.append(y)
    return order if len(order)==n else []   # [] = cycle

When: shortest path with weighted edges and no negatives.

import heapq
def dijkstra(graph, src):
    dist = {src: 0}; pq = [(0, src)]
    while pq:
        d, u = heapq.heappop(pq)
        if d > dist.get(u, 9e9): continue
        for v, w in graph[u]:
            if d + w < dist.get(v, 9e9):
                dist[v] = d + w
                heapq.heappush(pq, (d + w, v))
    return dist

When: "next greater/smaller element", largest rectangle, stock span — anything needing the nearest bigger/smaller neighbour.

def next_greater(nums):
    res = [-1]*len(nums); stack = []   # holds indices
    for i, x in enumerate(nums):
        while stack and nums[stack[-1]] < x:
            res[stack.pop()] = x
        stack.append(i)
    return res
